What is JamBridge?

JamBridge is the HL7v2-to-FHIR R4 integration gateway at the centre of the Jam clinical data platform. It sits between your existing hospital information systems — Epic, Cerner, MEDITECH, or any HIS that speaks HL7v2 — and a FHIR R4 data store, translating decades of HL7v2 message traffic into modern, queryable, consent-governed FHIR resources in real time.

It is not a middleware bus or an iPaaS. It is a purpose-built clinical integration engine that understands the semantics of HL7v2 messages — why a PID segment means a patient identity event, why an OBX with a LOINC code means a lab result that may require a drug safety check, why an RDE means a prescription that must be checked for drug-drug interactions across all branches before being written. That clinical understanding is baked into the 7-stage pipeline and cannot be bolted on after the fact.

🏗
Open source foundation
JamBridge is built on the AJ FHIR Platform — an Apache 2.0 open source infrastructure stack. The HAPI FHIR server, Auth Server, Consent Manager, MPI, and Facility Registry are all open source and auditable. JamBridge is the commercial integration engine that connects them into a working clinical pipeline.

Core responsibilities

JamBridge does seven things, in order, for every HL7v2 message it receives:

  1. Receive — raw HL7v2 bytes over MLLP, parse with HAPI HL7v2 Terser, stamp a transaction ID.
  2. Deduplicate — check MSH-10 control ID against a 24-hour window in hie_transaction_log. Replayed messages are ACK’d without re-processing.
  3. Enrich — resolve sending facility to a FHIR Organization via JamFR (ITI-90) and resolve patient to a golden record ID via JamMPI (ITI-119 $match).
  4. Transform — PID → Patient, PV1 → Encounter, OBX → Observation with LOINC codes validated and translated by JamTS.
  5. Govern — call AJ Consent Manager via JamConnect. If the patient has withheld consent, generate an HL7v2 NACK and stop. Fail-closed: NACK if consent service is unreachable.
  6. Persist — conditional create/update to HAPI FHIR. Circuit breaker protects HAPI from thundering-herd failures.
  7. Dispatch — FHIR subscription fan-out, outbound MLLP to LIS, JamConnect POST_WRITE events, DHIS2 surveillance pushes, ATNA audit.

The 7-stage clinical pipeline

Every message enters the same pipeline. The stages are sequential and non-configurable — this is deliberate. Consent must be checked before data is written. Identity must be resolved before transformation. A general-purpose bus cannot enforce this ordering without custom mediators that re-implement clinical logic for every deployment.

1
RECEIVE
MLLP reception + MSH-10 deduplication
Raw HL7v2 bytes arrive on a Camel MLLP consumer. MSH-10 is checked against hie_transaction_log. Duplicate messages within 24 hours are ACK’d without re-processing. The Hl7v2Parser extracts typed objects using the HAPI HL7v2 Terser.
2
ENRICH
Facility registry (ITI-90) + MPI $match (ITI-119)
FacilityCodeResolver calls JamFR on a 5-minute cache. MpiClientService sends Patient/$match to JamMPI — Jaro-Winkler name scoring, date-of-birth similarity, exact national ID matching. The golden record ID is stamped on the FHIR resource via Patient.link[refer].
3
TRANSFORM
HL7v2 → FHIR R4 + terminology translation
Hl7v2ToFhirMapper converts parsed HL7v2 to FHIR R4. OBX segments become Observations — JamTS validates LOINC codes via ConceptMap/$translate and federates to Snowstorm for SNOMED CT diagnosis codes.
4
GOVERN
Consent enforcement via AJ Consent (fail-closed)
JamConnectClient.checkConsent() calls AJ Consent Manager. Patient reference, actor, resource type, action (read/write), and purpose (TREAT / RESEARCH / PUBLIC-HEALTH) are checked. "permitted": false generates a NACK and stops the pipeline. Unreachable consent service also NACK’s — fail-closed by design.
5
PERSIST
HAPI FHIR conditional create/update
FhirClientService.conditionalUpdate() sends PUT /fhir/Patient?identifier=... to HAPI. A Resilience4j circuit breaker protects against HAPI downtime — open-circuit messages queue in SecondaryRetryQueue with exponential backoff, up to 10 attempts. ConsentEnforcementInterceptor on HAPI provides a second consent check for any SMART app queries.
6
DISPATCH
Subscription fan-out + JamConnect + downstream adapters
SubscriptionEventDispatcher evaluates 17 seeded FHIR subscriptions asynchronously. Matched channels fire outbound MLLP, FHIR REST, or SMS deliveries. JamConnect’s POST_WRITE event triggers IPS cache invalidation in JamSHR, CDS Hooks dispatch, and clinical alert escalation. ORU messages with notifiable LOINC codes (dengue, TB, malaria) fire DHIS2 surveillance pushes.
7
ACK
HL7v2 MSA|AA acknowledgement + ATNA audit
MSA|AA is returned to the sending system on the same MLLP connection. Transaction log is finalised. IHE ATNA AuditEvent is written asynchronously via RFC 5425 TLS syslog and as a BALP FHIR AuditEvent to HAPI. Audit writing never blocks the ACK path.

The Jam ecosystem

JamBridge is one component in a coordinated platform of 10+ products. Each is independently deployable, but designed to work together. The integration points between components are all standard FHIR R4 operations or REST APIs — nothing proprietary.

JamBridge vs OpenHIM

OpenHIM is the most widely deployed open source health interoperability middleware in low- and middle-income country deployments, built by the Health Architecture Lab at the University of Cape Town and used as the reference mediator platform for OpenHIE. JamBridge was built specifically to replace it in contexts where clinical depth, FHIR R4 nativity, and commercial-grade reliability are required.

Philosophy and architecture

The fundamental difference is architectural philosophy. OpenHIM is a general-purpose middleware bus — it routes messages through a mediator chain but has no inherent understanding of clinical content. Each mediator must implement its own parsing, transformation, and consent logic. OpenHIM does not know what an ADT A01 means, or that an OBX-8 = H requires different downstream action than OBX-8 = N.

JamBridge is a clinical integration engine. The pipeline is not configurable precisely because clinical workflows have a correct order. A general-purpose bus cannot enforce this ordering without custom mediators that re-implement clinical logic for every deployment.

⚖️
When OpenHIM is still the right choice
If your integration needs are primarily routing HL7v2 messages between systems without FHIR transformation, and you have the developer capacity to build and maintain custom mediators, OpenHIM remains a solid, free option with a large community. JamBridge is the right choice when you need real-time FHIR R4 transformation, cross-facility patient identity, consent enforcement, clinical safety rules, and a production-grade commercial SLA.

Migration path from OpenHIM

Most OpenHIM deployments follow a pattern: an MLLP mediator receives HL7v2, a transformation mediator converts to JSON or XML, and the output is forwarded downstream. JamBridge replaces all of this with a single service, but the migration does not need to be a cutover.

  1. Weeks 1–2: Deploy JamBridge alongside OpenHIM. Configure new MLLP ports. Begin mirroring a subset of traffic to JamBridge.
  2. Weeks 3–4: Validate FHIR output. Resolve any HL7v2 segment edge cases from your HIS vendor.
  3. Weeks 5–6: Redirect production MLLP traffic to JamBridge. Decommission OpenHIM mediators for migrated message types.
  4. Week 8+: Full cutover. OpenHIM decommissioned.

JamBridge in a large hospital chain

The hospital chain deployment is where JamBridge’s cross-facility architecture shows its value most clearly. A single patient attends multiple branches. Their medication is prescribed at one branch, their lab results are at another, their imaging is at a third. Without a unified golden record and a cross-facility FHIR store, clinicians at each branch see only a fragment of the patient’s history — and safety events like drug-drug interactions across branches are invisible.

Network topology

In a typical 3–10 branch chain, every branch runs its own HIS. JamBridge is deployed as a central service (or cluster for high availability) that all branches connect to over dedicated MLLP TCP sockets. FHIR REST connections from private-network hospitals use SMART Bearer tokens issued by the AJ Auth Server.

Cross-branch patient identity — the golden record

The hardest problem in a hospital chain deployment is not the integration — it is the identity. A patient attends Branch 01 as MRN12345. The same patient attends Branch 03 as PRIV-9901. Without a Master Patient Index, these are two different patients in the FHIR store, and JamGuard cannot know that the Warfarin prescribed at Branch 01 and the Aspirin prescribed at Branch 03 belong to the same person.

JamMPI resolves this on every message using three strategies in combination: national ID exact match (earns CERTAIN grade, score ≥ 0.95), Jaro-Winkler name + date-of-birth similarity (earns PROBABLE above a configurable threshold), and a blocking strategy that pre-filters candidates by first-3-chars of family name and birth year to keep comparison space manageable.

The golden record ID is stamped on every FHIR Patient resource via Patient.link[type=refer] and a custom extension. Every downstream system that queries HAPI for this patient by golden ID gets all records from all branches.

Full data flow walkthrough

Here is what happens when Branch 01’s MEDITECH HIS admits a patient who was previously seen at Branch 03:

  1. MEDITECH sends ADT^A01 to JamBridge on MLLP :2575. MSH-10 BR01-001 is checked — not a duplicate.
  2. JamFR resolves MSH-4 BRANCH-01 to Organization/org-br01-001 (cached 5 min).
  3. JamMPI runs Patient/$match. Finds CERTAIN match to GOLDEN-00441 — the same patient seen at Branch 03 under a different local MRN.
  4. JamTS translates local lab codes to LOINC and SNOMED CT diagnosis codes via Snowstorm.
  5. AJ Consent Manager: patient GOLDEN-00441, actor org-br01-001, resource Patient, action write, purpose TREAT. "permitted": true.
  6. HAPI FHIR: PUT /fhir/Patient?identifier=MRN12345. ConsentEnforcementInterceptor fires. Patient/MRN123456 version 3 written.
  7. 3 outbound subscription channels fire. JamConnect POST_WRITE invalidates JamSHR IPS cache for GOLDEN-00441.
  8. Dr. Smith opens JamPHR. GET /fhir/Patient/GOLDEN-00441/$summary returns an IPS from Branch 01, Branch 03, and the central lab — medications from both branches, allergies recorded at Branch 03, imaging from Branch 02’s PACS.
  9. Dr. Smith orders Warfarin 5mg. RDE^O11 arrives on :2580. JamGuard fetches all active MedicationRequests for GOLDEN-00441 across all branches. Finds Aspirin 100mg at Branch 03. NLM RxNorm: MAJOR interaction. Alert fires before the prescription is written.

Cross-branch safety rules in practice

JamGuard’s power in a hospital chain context is that it sees the complete medication list — not just what was prescribed at the current branch. The four active rules in a typical chain deployment:

  • DRUG_DRUG_INTERACTION: On every MedicationRequest write, fetches all active MedicationRequests for the golden patient ID across all branches and sends the RxNorm CUI list to the NLM RxNorm Interaction API. MAJOR interactions block and alert the prescriber. MODERATE interactions create a warning card without blocking.
  • DRUG_ALLERGY: Checks new MedicationRequest against all AllergyIntolerance resources for the golden patient across all branches.
  • MISSED_FOLLOWUP: Hourly scan for Observations with OBX-8 = H (High) or L (Low) from the last 72 hours for which no follow-up Encounter has been created.
  • DUPLICATE_PRESCRIPTION: Checks whether a new MedicationRequest is in the same ATC drug class as a medication already active from another branch within 30 days.
